# Mini CRM

A production-ready Mini CRM built with **Laravel 13**, **React** (via Inertia.js), and **MySQL**. It covers the full lead lifecycle — creation, assignment, notes, status management, and automated conversion tracking through queued jobs.

The backend exposes a RESTful JSON API secured with **Laravel Sanctum**. The frontend is a React + TypeScript SPA served by Laravel, with role-aware UI and MUI components.

---

## Quick Start

> **Prerequisites:** PHP 8.2+, Composer 2.x, Node 18+, MySQL 8.x

### Option A — Import the pre-built database (fastest)

If you want to skip migrations and get started with demo data already loaded:

```bash
# 1. Clone the repo
git clone <your-repo-url> mini-crm
cd mini-crm

# 2. Install PHP and Node dependencies
composer install
npm install

# 3. Set up your environment file
cp .env.example .env
php artisan key:generate

# 4. Create the MySQL database
mysql -u root -p -e "CREATE DATABASE mini_crm CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# 5. Import the pre-built database dump
mysql -u root -p mini_crm < database/mini-crm-db.sql

# 6. Update your .env with your MySQL credentials
#    (see Environment Configuration section below)

# 7. Build frontend assets
npm run build

# 8. Start the server
php artisan serve
```

Visit **http://localhost:8000** — login with the demo credentials below.

## Queue Setup

The app uses the **database** queue driver — no Redis or any external service needed. To process background jobs (lead conversion tracking), run:

```bash
php artisan queue:work
```

Keep this running in a separate terminal alongside `php artisan serve`. In production, use Supervisor to keep the worker alive automatically.

When a lead status is changed to **Converted**, a `LeadConvertedJob` is dispatched. It does two things:
- Creates a record in the `lead_conversions` table (lead ID, name, converted by, timestamp)
- Writes a structured log entry to `storage/logs/laravel.log`

**How to authenticate in Swagger UI:**
1. Call `POST /api/auth/login` with your credentials — copy the `token` from the response
2. Click the **Authorize** 🔒 button at the top of the page
3. Paste the token (without the word "Bearer") and click Authorize
4. All subsequent requests will now send the token automatically

## Authentication Flow

The API uses Sanctum token-based authentication:

```
POST /api/auth/login      → returns a bearer token
GET  /api/auth/me         → returns the authenticated user
POST /api/auth/logout     → revokes the current token
```

Include the token on every protected request:
```
Authorization: Bearer <your-token>
```

## Frontend (Bonus)

The frontend is a React + TypeScript SPA served by Laravel — no separate Node server. All 5 required screens are implemented:

| Screen | Route |
|---|---|
| Login | `/login` |
| Leads Listing (search + status filter + pagination) | `/leads` |
| Add Lead | `/leads/create` |
| Edit Lead | `/leads/{id}/edit` |
| Lead Details (notes timeline + status updater) | `/leads/{id}` |

There's also a **Dashboard** at `/dashboard` with live stats (total, new, contacted, converted leads).

Role-aware UI: the "Delete" button only appears for admin users. The "New Lead" button is visible to all authenticated users. Authorization is always enforced on the backend — the UI just reflects permissions.

## Database Schema

| Table | Purpose |
|---|---|
| `users` | Authenticated users with `role` (admin/user) |
| `leads` | Core lead entity — status, assignee, creator, soft-delete |
| `lead_notes` | Notes per lead, ordered chronologically |
| `lead_conversions` | Written by queue job on conversion |
| `personal_access_tokens` | Sanctum API tokens |
| `jobs` | Laravel database queue table |

Indexes on `leads.status`, `leads.assigned_to`, and `lead_notes.(lead_id, created_at)` for fast filtering and retrieval.

## Assumptions & Design Decisions

**Integer PKs over UUIDs** — UUIDs add index overhead without benefit at this scale. Integers are faster and simpler.

**No Repository Pattern** — The Service Layer backed by Eloquent is already a clean separation of concerns. Repositories would be indirection with no real benefit here.

**Database queue driver** — Zero external dependencies. Switching to Redis in production is a one-line change: `QUEUE_CONNECTION=redis`.

**Both Option A and B for conversions** — `LeadConvertedJob` writes to `lead_conversions` AND logs the event, satisfying the BRD requirement and providing operational visibility.

**Policy `before()` hook** — Admins bypass all policy checks through a single `before()` method, avoiding repetitive `if ($user->isAdmin()) return true;` in every policy method.

**Soft deletes on leads** — Rather than permanently destroying lead records, deletes set a `deleted_at` timestamp. This preserves data integrity and allows recovery if needed.

**Standard users can create leads** — The BRD doesn't explicitly restrict this. They can create leads but can only view and manage leads assigned to them.

## Known Limitations

- No email verification flow
- No rate limiting on the login endpoint (would add `throttle:5,1` middleware in production)
- `phone` is stored as a plain string — no E.164 format validation
- Offset-based pagination (not cursor-based) — fine for this scale
